# 多智能体 AI 平台 — 架构设计与演进

## 一、项目概述

基于 FastAPI 的五层解耦式智能体协作平台：**L1 LLM 基座 + L2 RAG 知识库 + L3 MCP 工具总线 + L4 ReAct 决策循环 + L5 React 前端**。集成 LangGraph 编排、LLM 轮询池熔断、LoRA 本地推理、流式响应、响应缓存。目标：生产级可用，高并发下稳定运行。

---

## 二、五层架构

完整调用链路：**L5 (React) → L4 (ReAct) → Action → L3 (MCP) → L2 (RAG) → L1 (LLM) → 原路返回**。

```
L5  React SPA (Vite + TypeScript + TailwindCSS)
    │  SSE streaming / Auth / KB manager / 限流面板
    ▼
L4  ReAct 决策层
    │  _run_react_loop(): Function Calling / Thought → Action → Observation → ... → Final Answer
    │  tool_agent (全部工具) / knowledge_agent (rag_search+web_search)
    │  lora_agent (降级路径)
    ▼   Action: rag_search | web_search | rate_stats | session_info
L3  MCP 统一通信协议
    │  MCPToolRegistry: register / list_tools / call_tool
    │  参数校验 / 调用日志 / 权限管控
    ▼
L2  RAG 知识库
    │  BGE small embedding (512维) + BM25 混合检索
    │  语义分块 / query 重写 / sigmoid 融合
    ▼
L1  LLM 基座
    │  DeepSeek API (3 Key 轮询 + 断路器) / Qwen2.5-3B + LoRA
    └──────────────────────────────────────────────
```

### L1 — 模型基座层

**核心引擎**：DeepSeek API（3 Key 轮询 + 断路器）为主，Qwen2.5-3B + LoRA 适配器为辅。

- 所有问答、思考、文本生成的基础推理
- LoRA 适配器（世界杯助手）基于 34 条 Q&A 训练，10 epoch，loss 2.261→1.254
- **不参与工具通信**，属于模型内部优化
- 断路器三态（CLOSED → OPEN → HALF_OPEN），全部熔断时降级

### L2 — 知识数据层

**数据底座**：文档向量化 + BM25 混合检索。

- 支持文档：`.txt` / `.md` / `.pdf`
- 当前知识库：17 块（4 世界杯 + 13 体育通用）
- BGE 本地嵌入（bge-small-zh-v1.5，512 维），离线运行
- 语义分块：Markdown 标题 → 段落 → 句子边界
- **只能通过 L3 MCP 的 `rag_search` 工具访问**，各节点禁止直连

### L3 — MCP 统一通信协议

**标准化总线**：连接所有 Agent 与外部工具的中间层。

| 工具 | 描述 | 参数 |
|------|------|------|
| `rag_search` | 搜索本地知识库 | `query` + `k` |
| `web_search` | 联网搜索实时信息 | `query` |
| `rate_stats` | 查看限流和熔断状态 | 无 |
| `session_info` | 查看会话信息 | `session_id` |

特性：
- 统一参数校验、权限管控、调用日志
- 新增数据源直接注册，不改 Agent 核心代码
- 所有 `Observation` 通过 MCP 返回给 ReAct 循环
- 自研轻量实现（~60 行），零外部依赖

### L4 — ReAct 智能体决策层

**系统调度中心**：支持双路径工具调用——优先使用 OpenAI Function Calling API，不支持的模型（本地 LoRA）自动降级为文本格式 ReAct。共享 `_run_react_loop()` 函数。

```
Thought: 用户问巴西世界杯成绩，需要查知识库
Action: rag_search
Action Input: {"query": "巴西 世界杯 成绩"}
Observation: [知识库返回巴西5次冠军...]
Thought: 信息足够，可以直接回答
Final Answer: 巴西共获得5次世界杯冠军...
```

可用工具列表通过 MCP 注册中心动态生成，每个节点限定可访问的工具范围：

| 节点 | 可用工具 |
|------|----------|
| `tool_agent` | 全部工具 |
| `knowledge_agent` | `rag_search` + `web_search` |
| `lora_agent`（降级时） | `rag_search` + `web_search` |

最大 5 步推理，格式异常或超步数自动兜底。

### L5 — 前端应用层

**React SPA**（`frontend/`），Vite + TypeScript + TailwindCSS v4，功能：
- 对话交互（SSE 流式）
- 知识库管理（上传/删除文档）
- 限流状态展示
- 会话管理
- 访客模式 / 登录注册

## 三、模块设计与演进

### 3.1 Agent Graph（智能体图）

**方案**：LangGraph `StateGraph` + `TypedDict` + `CompiledStateGraph`

```
intent ──┬── code_agent ────── supervisor ──┬── code_agent (最多 3 轮)
         ├── knowledge_agent ─supervisor────┤── knowledge_agent
         ├── data_agent ───────supervisor───┤── data_agent
         ├── tool_agent ────────────────────┤── merger → END
         ├── chat_agent ────────────────────└── END
         └── lora_agent ────────────────────── END
```

**节点说明**：

| 节点 | 功能 | 超时 | 后续 |
|------|------|------|------|
| intent | LLM 意图分类 / 关键词直投 lora | 10s | 7 路分支 |
| code_agent | 编程/代码助手 + MCP 工具列表 | 60s | supervisor |
| knowledge_agent | ReAct 循环（rag_search + web_search） | 60s | supervisor |
| chat_agent | 日常闲聊 + 搜索 | 30s | END |
| data_agent | 数据分析 + MCP 工具列表 | 60s | supervisor |
| tool_agent | ReAct 循环（全部工具） | 30s | END |
| lora_agent | LoRA 推理 → 降级 ReAct-MCP | 30s | END |
| supervisor | 质检 + 重分类 + 循环控制 | 10s | 条件边 |
| merger | 合并并行结果 | 15s | END |

**核心代码**：
```python
class AgentState(TypedDict):
    messages: List[Dict[str, str]]
    intent: str
    loop_count: int
    task_complete: bool
    final_answer: str
    request_id: str
    user_id: str
    session_id: str
    parallel_results: Dict[str, str]

graph = StateGraph(AgentState)
graph.add_node("intent", intent_node)
graph.add_node("code_agent", code_agent)
# ...
compiled = graph.compile()
result = await compiled.ainvoke(initial_state)
```

**演进**：
- V1：自研 FSM + 死循环检测
- V2：LangGraph，直连 RAG/搜索
- V3：ReAct 循环 + MCP 解耦

### 3.2 LLM Pool（API Key 轮询池）

**核心问题**：DeepSeek API 单 key 有频率限制（RPM/TPM），单 key 故障整站不可用。

**方案**：3 个 key 轮询 + 熔断器自动切换。

```
CLOSED（正常）─连续失败≥阈值─→ OPEN（熔断，冷却 10s→120s）
OPEN（冷却完）──────────────→ HALF_OPEN（探针）
HALF_OPEN（成功）───────────→ CLOSED
HALF_OPEN（失败）───────────→ OPEN（重置计时）
```

`_call_llm` 先用轮询选主实例，失败后遍历剩余健康实例，全部熔断降级。

**演进**：
- V1：单 key 直连
- V2：circuit breaker
- V3：3 key 轮询 + 熔断感知
- V4：`_call_llm` 主备切换

### 3.3 RAG 知识库

**设计**：文档 → 分块（语义边界） → BGE small embedding → BM25+向量混合检索。

**混合检索**：
```
BM25(score) = sigmoid(BM25_raw) * 0.4
Vector(score) = cosine_similarity * 0.6
Final = BM25 + Vector
```

**查询重写**：LLM 将用户口语化问题改写为检索友好的关键词组合。

**演进**：
- V1：jieba + TF-IDF
- V2：BGE embedding + numpy 向量检索
- V3：BM25+向量混合 + 语义分块 + query 重写

### 3.4 会话历史

**设计**：Redis → 内存二级降级，TTL=3600s（1 小时）。

```
get_history(session_id):
    if redis_reachable: return redis.get(session_id)
    if memory_store[sid].ts < TTL: return memory_store[sid].msgs
    return []

save_history(session_id, messages):
    recent = messages[-20:]
    redis.setex(key, TTL, json(recent))  # Redis 自动过期
    memory_store[sid] = {msgs: recent, ts: now}  # 双重写入
```

**演进**：
- V1：仅 Redis，不可用抛异常
- V2：try/except 降级内存
- V3：发现 redis-py 8.0 socket_connect_timeout 失效，每次等 25.6s
- V4：TCP 探活 `_redis_reachable()` + 模块级缓存，不可用 < 2ms

### 3.5 LLM 响应缓存

**设计**：内存 LRU 缓存，key = SHA256(system_prompt + user_message + temperature)

```
缓存命中：跳过 BGE + 搜索 + LLM → 5ms 返回
缓存未命中：RAG 注入 → LLM 调用 → 缓存（key 用原始 prompt）
```

**关键决策**：key 用原始 prompt（不含 RAG/搜索）。保证相同问题命中同一 key；回答本身已包含当时注入的上下文信息。

**参数**：TTL=3600s，容量=1000 条。

**演进**：
- V1：无缓存
- V2：缓存放 RAG 注入后（命中也要等 BGE）
- V3：缓存放最前面，key 用原始 prompt

### 3.6 MCP 工具总线（Model Context Protocol）

**设计**：自研轻量 `MCPToolRegistry`，统一管理所有 Agent 可调用的外部工具。

```python
class MCPToolRegistry:
    def register(self, name, description, handler, parameters)
    def list_tools(self) -> List[Tool]
    async def call_tool(self, name, arguments) -> str
```

**已注册工具**：

| 工具 | 层 | 描述 |
|------|----|------|
| `rag_search` | L2→L3 | 搜索本地知识库，返回最相关文档片段 |
| `web_search` | L3 | 联网 Bing/Baidu 并发搜索 |
| `rate_stats` | L3 | 查看限流和熔断状态 |
| `session_info` | L3 | 查看会话历史数量 |

**工作流**：
1. L4 ReAct 节点输出 `Action: rag_search` + `Action Input: {"query": "..."}`
2. L3 `mcp_registry.call_tool("rag_search", {...})` 执行
3. L2 RAG 引擎检索 → 结果返回 L4 Observation
4. ReAct 继续推理直到 Final Answer

### 3.7 ReAct 决策循环

**设计**：共享 `_run_react_loop()` 函数，各节点仅指定可用工具列表。

```python
async def _run_react_loop(
    user_message: str,
    *,
    node_name: str = "react",
    tool_names: Optional[List[str]] = None,
    max_steps: int = 5,
) -> str:
```

**节点工具隔离**：

| 节点 | 可用工具 |
|------|----------|
| `tool_agent` | 全部 4 工具 |
| `knowledge_agent` | `rag_search` + `web_search` |
| `lora_agent`（降级） | `rag_search` + `web_search` |

**函数调用（Function Calling）**：
- 工具自动转为 OpenAI 兼容的 `tools` 参数传递给 LLM
- LLM 返回 SDK 解析的 `tool_calls` → 直接执行，无需正则解析
- 本地/LoRA 模型不支持函数调用时，自动降级为文本格式

**文本格式（降级）**：
```
=== 调用工具 ===
Thought: 推理过程
Action: tool_name
Action Input: {"params"}

=== 已有答案 ===
Thought: 推理过程
Final Answer: 最终回答
```

**演进**：
- V1：工具调度器单次调用（LLM 选择工具 → 执行 → 返回）
- V2：tool_agent 独立实现 ReAct 循环
- V3：共享 `_run_react_loop`，所有节点统一 ReAct-MCP 路径
- V4：支持 Function Calling API 优先，文本 ReAct 降级

### 3.8 LoRA 本地推理

**设计**：Qwen2.5-3B + PEFT LoRA 适配器，RTX 4060 8GB 运行。

```
体育/世界杯关键词 → intent_node → lora_agent:
  1. RAG context 注入（直连 L2）
  2. LoRA 推理（L1 模型层）
  3. 质量门禁（长度<10 / stale → 降级 ReAct-MCP）
```

**训练**：34 Q&A 体育解说风格，10 epoch，96s，loss 2.261→1.254。

### 3.9 限流器（Token Bucket）

**设计**：内存令牌桶，全局 + 每用户两级。

- GLOBAL_RATE=100/s, GLOBAL_CAPACITY=200
- USER_RATE=10/s, USER_CAPACITY=20
- 令牌不足 → 429 Too Many Requests

## 五、项目亮点（面试用）

1. **五层解耦架构** — L1 LLM + L2 RAG + L3 MCP + L4 ReAct + L5 React，每层职责清晰，数据流单向
2. **MCP 强制解耦** — 所有数据源统一注册，节点禁止直连，新增数据源只需注册工具
3. **共享 ReAct 循环** — `_run_react_loop()` 通用函数，节点仅指定 tool_names，避免重复代码
4. **LangGraph + ReAct 结合** — 图控制节点生命周期，ReAct 控制工具调用逻辑
5. **熔断器 + 缓存 + 降级** — 3 层防御保障单点故障不影响整体可用性

## 七、架构演进历史

| 阶段 | 架构 | 特点 |
|------|------|------|
| v1 | 直接 LLM 调用 | 无分层，无缓存，无熔断 |
| v2 | 自实现 FSM + RAG 直连 | 简单的 if/elif 状态机 |
| v3 | LangGraph DAG + RAG 直连 | StateGraph 编排， supervisor 质检 |
| v4 | ReAct 循环 + RAG 直连 | tool_agent 独立 ReAct，其他节点直连 RAG |
| **v5 (当前)** | **ReAct → MCP → RAG** | 五层解耦，所有数据走 MCP，共享 ReAct 循环 |
