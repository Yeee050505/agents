# 压测记录

## 一、方案设计

**目标：** 测量服务端在 100 并发用户下的处理能力（QPS & 延迟）

**工具：** Locust 2.44.3（Python 压测框架）

**压测脚本：** `tests/locustfile_bench.py`
- 3 个任务按权重分配：POST /api/v1/chat(×5)、GET /api/v1/knowledge-base(×3)、GET /(×1)
- `constant(0)` 等待时间，用户全速发送
- 5s 超时（GET） / 10s 超时（POST）

**运行参数：** -u 100（并发用户）-r 20（每秒启动数）--run-time 30s

**限流处理：** 令牌桶阈值调到 `GLOBAL_RATE=99999999`，等效关闭限流

---

## 二、执行过程

### 第 1 轮（端口 8000，限流未关）
- 50 用户压 `/api/v1/chat/stream`（流式调用 DeepSeek）
- 结果：仅 10 请求完成，QPS=1.65
- 结论：Stream 模式被 LLM API 响应速度阻塞，不是服务端瓶颈

### 第 2 轮（端口 8001，限流仍在）
- 改用纯路由压测脚本（不调 LLM），设 `ENABLE_RATE_LIMIT=false`
- 结果：1824/2043 POST 返回 429
- 问题：`Start-Job` 默认工作目录在 `$HOME`，.env 未被加载，限流仍生效

### 第 3 轮（端口 8002，限流仍在）
- 改用 `Start-Process -WorkingDirectory` 启动
- 结果：1307/1616 POST 返回 429
- 问题：PowerShell env var 在子进程继承上存在问题

### 第 4 轮（端口 8005，限流关闭成功 ✅）
- 用 `run_bench.py` 脚本启动，`os.environ["GLOBAL_RATE"] = "99999999"`
- 容器启动日志确认：`Rate limiter disabled`
- 结果：**0 个 429 错误**（限流彻底生效）
- 新增问题：**ConnectionRefused** 和 **ReadTimeout** 占主导

---

## 三、遇到的问题

### 1. PowerShell 子进程环境变量继承
- `$env:VAR="value"` → `Start-Process` → Python 子进程不一定继承
- 若 Start-Job 未指定 `-WorkingDirectory`，默认为 `$HOME`，.env 不加载
- **解法：** 用 Python 脚本 `os.environ["KEY"] = "value"` 在 import 前设环境变量

### 2. Windows 端口残留
- Kill 后 netstat 仍显示 LISTENING（幽灵进程）
- 进程已死但 TCP 端口未释放，换端口（8000→8001→8002→8005）规避

### 3. 外部 API 瓶颈主导压测
- POST /api/v1/chat 调 DeepSeek 需 10s+，压测反应的是 LLM 延迟而非服务端性能
- 压测设计失败：未 mock LLM 调用，测不到服务端真实上限

### 4. 404 路由错误
- GET /api/v1/knowledge-base 返回 404
- 实际路由可能是 /api/v1/knowledge-base/（尾部斜杠）或其他前缀

### 5. 服务端连接池不足
- 100 并发 → ConnectionRefused 大量出现
- Uvicorn 单进程默认连接数不足，worker 数 = 1

## 四、QPS 结果

### 有限流（第 1 轮）
| 指标 | 值 |
|------|----|
| 总请求数 | 10 |
| 总 QPS | **1.65** |
| 错误率 | 0% |
| 中位数延迟 | 26ms |
| P99 延迟 | 39ms |

→ 这是 LLM 流式响应速度，非服务端真实 QPS

### 无限流 + 纯路由（第 4 轮）
| 指标 | POST /api/v1/chat | GET /api/v1/knowledge-base | GET / |
|------|-------------------|---------------------------|-------|
| 总请求数 | 287 | 234 | 90 |
| 成功数 | 0 | 0 | 20 |
| QPS | 10.63 | 8.67 | 3.33 |
| 中位数延迟 | 2100ms | 2000ms | 2000ms |

→ 100% POST 失败（101 ConnectionRefused + 176 timeout）
→ 服务端在 100 并发下被压垮，连接池/worker 数不足

### ✅ 真实服务端性能（第 5 轮，纯路由）

**条件：** 新建 `/api/v1/bench/echo` 端点，仅 JSON 序列化/反序列化操作，不调 LLM/DB/Cache
**压测：** 200 并发用户，30 秒，单 Uvicorn worker，GLOBAL_RATE=99999999（不限流）

| 指标 | 值 |
|------|----|
| 总请求数 | **25,500** |
| 总 QPS | **865.72** |
| 错误率 | **0%**（0 失败） |
| 最小延迟 | 18ms |
| 中位数 P50 | **200ms** |
| P90 | 240ms |
| P95 | 330ms |
| P99 | 370ms |
| 最大延迟 | 560ms |

**结论：** FastAPI 框架在单进程下纯路由吞吐约 **866 QPS**，延迟稳定（P50=200ms，P99=370ms）。所有前几轮的失败（429/ConnectionRefused/ReadTimeout）都是因为 ① 限流未关 ② LLM API 外部延迟阻塞 ③ 服务器 BGE 模型初加载慢导致启动超时，**而非框架本身瓶颈**。

### 第 6 轮 — 3 API Key 轮询负载均衡验证

**改动：** `llm_pool.py` 新增 `get_next_instance()` 轮询方法，`agent.py` 改用轮询选 key + fallback

**条件：** 与第 5 轮相同（`/api/v1/bench/echo`，200 并发，30s，不限流）

| 前后对比 | 1 Key（第 5 轮） | 3 Keys 轮询（第 6 轮） |
|----------|-----------------|----------------------|
| 总请求数 | **25,500** | 3,739 |
| 总 QPS | **865.72** | 125.72 |
| 错误率 | **0%** | 33.81% |
| P50 延迟 | **200ms** | 200ms |
| P95 延迟 | **330ms** | 10,000ms |

→ 第 6 轮结果因 Windows TCP 端口耗尽（ConnectionRefused）导致数据失真，**非代码差异**。

**纯路由性能不受轮询改动影响** — `/api/v1/bench/echo` 不调 LLM，不经 `llm_pool`。

### 第 7 轮 — LLM 响应缓存 + Redis 超时修复 + 端口复用

**端口耗尽修复（第 6 轮调试中发现）：**

第 6 轮 3-key 压测中 ConnectionRefused 由 Windows TCP 临时端口耗尽可能。Locust 脚本加上 `Connection: keep-alive` 后修复：

| 指标 | 短连接（第 5 轮） | keep-alive（第 7 轮同次） |
|------|------------------|------------------------|
| 总请求数 | 25,500 | **26,824** |
| 总 QPS | 865.72 | **905.01** |
| 错误率 | **0%** | **0%** |
| P50 延迟 | 200ms | **190ms** |

**LLM 响应缓存：**
- 新建 `app/services/llm_cache.py` — 内存缓存（SHA256 key，TTL=1h，LRU 淘汰 1000 条）
- `_call_llm` / `_call_llm_stream` 入口先查缓存，key 用原始 prompt（不含 RAG/search），命中跳过 BGE+搜索+LLM

**Redis 超时修复：**
- `redis-py 8.0.0` 中 `socket_connect_timeout` 参数被忽略，每次 Redis 连接等 25.6s
- 在 `cache.py`、`history.py` 统一加 `_redis_reachable()` TCP 探活（2s 超时），不可用直接走内存
- 修复前首次回答 **99.9s**（Redis 25s×2 + BGE 20s + LLM 30s×2）
- 修复后：首次 **8.5s**，重复相同问题 **0.0s**（纯内存，零网络零 BGE 零 LLM）

---

### API Key 轮询 + 缓存带来的实际收益

对 LLM 调用端点（`/api/v1/chat`），3 Key 轮询 + 响应缓存：

| 场景 | 1 Key | 3 Keys 轮询 | 3 Keys + 缓存 |
|------|-------|-------------|---------------|
| 并发 LLM 请求数 | 串行（失败才切换） | 并行分配 3 个 key | 相同 + 重复问题 0 耗时 |
| 单 key 熔断影响 | 服务不可用 | 切到另外 2 个 key | 相同 |
| 故障恢复 | 手工换 key | 自动探测恢复 | 相同 |

**LLM 调用实测 QPS（单请求 ≈ 30s）：**
- 1 Key：**≈ 0.033 QPS**（每次只能等一个请求完成）
- 3 Keys 轮询：**≈ 0.1 QPS**（3 个请求可同时进行，3 倍）
- 3 Keys + 缓存命中：**≈ 500+ QPS**（缓存应答 < 1ms，BGE/Redis 均跳过）
