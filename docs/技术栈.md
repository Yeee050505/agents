# 多智能体平台 — 技术栈与架构设计

## 一、项目定位

基于 **LangGraph + DeepSeek + MCP + ReAct** 构建的五层解耦式智能体编排引擎，支持意图路由、ReAct 决策循环、MCP 工具总线、RAG 知识库、LoRA 本地推理、流式输出、高并发防御，是一个生产级 AI Agent 中台原型。

---

## 二、五层架构

```
L5 前端 (React SPA)
  ↓
L4 ReAct 决策 (Thought → Action → Observation → ... → Final Answer)
  ↓  Action: rag_search / web_search
L3 MCP 统一通信协议 (工具注册中心、参数校验、调用日志)
  ↓
L2 RAG 知识库 (向量+BM25 混合检索, BGE 本地嵌入)
  ↓
L1 LLM 基座 (DeepSeek API 3 Key 轮询+断路器 / Qwen2.5-3B+LoRA)
```

### L1 — 模型基座层
- **DeepSeek API**: 主推理引擎，3 Key 轮询，断路器三态（CLOSED→OPEN→HALF_OPEN），指数退避 10s→120s
- **Qwen2.5-3B + LoRA**: 本地 GPU 推理层，RTX 4060 8GB，bfloat16，世界杯助手适配器（34 Q&A，10 epoch）

### L2 — 知识数据层
- **BGE small 本地嵌入**（bge-small-zh-v1.5, 512维）：离线运行，首次懒加载
- **BM25 + 向量混合检索**：权重 0.4 + 0.6，sigmoid 归一化
- **语义分块**：Markdown 标题 → 段落 → 句子边界
- **当前知识库**：2 文档 17 块（世界杯 + 体育通用），支持 .txt / .md / .pdf

### L3 — MCP 统一通信协议
- **自研 MCPToolRegistry**（~60 行）轻量工具注册中心
- **已注册工具**：

| 工具 | 描述 | 参数 |
|------|------|------|
| `rag_search` | 搜索本地知识库 | query + k |
| `web_search` | 联网搜索（Bing/Baidu 并发） | query |
| `rate_stats` | 查看限流和熔断状态 | 无 |
| `session_info` | 查看会话信息 | session_id |

- **强制约束**：所有外部数据访问必须走 MCP，节点禁止直连 RAG/搜索

### L4 — ReAct 决策层
- **共享 `_run_react_loop()` 函数**：Thought → Action → Observation → ... → Final Answer
- **节点工具隔离**：

| 节点 | 可用工具 |
|------|----------|
| `tool_agent` | 全部 4 工具 |
| `knowledge_agent` | `rag_search` + `web_search` |
| `lora_agent`（降级） | `rag_search` + `web_search` |

- 最大 5 步推理，格式异常或超步数自动兜底

### L5 — 前端应用层
- **React SPA**（Vite + TypeScript + TailwindCSS v4）
- SSE 流式渲染、知识库管理、限流面板、会话管理
- 开发模式 `/api` proxy → `localhost:8000`，免 CORS 配置

## 三、核心技术点（面试重点）

### 1. 多智能体编排（LangGraph）

```
intent ──┬── code_agent ────── supervisor ──┬── code_agent
         ├── knowledge_agent ─supervisor────┤── knowledge_agent  (最多 3 轮)
         ├── data_agent ───────supervisor───┤── data_agent
         ├── tool_agent ────────────────────┤── merger → END
         ├── chat_agent ────────────────────└── END
         └── lora_agent ────────────────────── END
```

- **LangGraph StateGraph**：TypedDict 状态，节点返回 dict 更新
- **意图分类**：temperature=0.1 LLM 分类为 code/knowledge/chat/tool/data/lora
- **Supervisor 重分类**：检测到 `不知道/我不清楚/无法回答` → 换意图重试
- **并行扇出**：Send() API 让 code/knowledge/data 并行执行，merger 合并

### 2. 知识库 RAG

**混合检索流程**：
```
用户提问 → query 重写（LLM） → BGE 嵌入 → 向量检索(0.6) + BM25(0.4)
→ sigmoid 融合 → Top-K 注入 prompt
```

**文本预处理**：去乱码、空行、全半角统一、无效行过滤、URL 删除、词频统计

**亮点**：
- 语义分块保证切分边界合理性
- 查询重写提升检索命中率
- BM25 零 GPU 开销，适配纯 CPU 环境

### 3. ReAct 决策循环

```python
# 通用 ReAct 函数，节点仅需指定可用工具列表
answer = await _run_react_loop(
    content,
    node_name="knowledge_agent",
    tool_names=["rag_search", "web_search"],
)
```

**循环流程**（双路径）：
1. **Function Calling 路径（优先）**：LLM 返回 SDK 解析的 `tool_calls` → 直接执行工具
2. **文本 ReAct 路径（降级）**：LLM 输出文本格式 `Action: tool_name` + `Action Input` → 正则解析 → 执行工具
3. 工具结果作为 `Observation` 追加到对话 → 继续循环
4. 最终输出自然语言回答

**亮点**：
- 工具自动转为 OpenAI Function Calling schema，零手动适配
- 不支持函数调用的模型自动降级为文本 ReAct
- LangGraph 显式图结构 + 节点级工具隔离 + 超时/重试/降级

### 4. LLM API 密钥池

```python
class LLMInstance:
    api_key       # 密钥
    state         # closed / open / half_open
    failures      # 连续失败计数
    cooldown      # 指数退避: 10s → 20s → 40s → 80s → 120s
```

- 多 Key 轮询：`get_healthy_instances()` 只返回未熔断的实例
- 全部熔断 → `is_degraded` → 返回友好降级文案
- `_call_llm` 先用轮询选主实例，失败后遍历剩余健康实例

### 5. 流式输出（SSE）

```
后端: ChatOpenAI.astream() → StreamingResponse(text/event-stream)
  → run_agent_stream → graph.astream() 节点级事件发射
前端: fetch + ReadableStream → 逐 token 更新气泡

格式:
data: {"token": "你好"}
data: {"token": "，"}
...
data: {"intent": "knowledge", "session_id": "xxx"}
data: [DONE]
```

**效果**：1s 内出现第一个字，逐字输出。

### 6. LLM 响应缓存

```
检查放 RAG 之前：
  key = SHA256(原始 system_prompt + user_message + temperature)
  缓存命中 → 直接返回，跳过 BGE + 搜索 + LLM（5ms）
  缓存未命中 → RAG 注入 → LLM 调用 → 缓存结果
```

**为什么不缓存 RAG 注入后的 prompt**：BGE 搜索结果每次可能不同，注入后 key 波动导致命中率为 0。

### 7. MCP 工具总线

```
MCPToolRegistry:
    register(name, description, handler, parameters)   ← 注册工具
    list_tools() → Tool[]                               ← 暴露给 Agent
    call_tool(name, arguments) → str                    ← 执行工具
```

**为什么自研不用 MCP SDK**：远程 MCP Server 需要独立进程管理，当前所有工具在本地即可完成，自研 60 行零依赖。

### 8. Redis 降级策略

- **TCP 探活**：`asyncio.open_connection(host, port, timeout=2)` 提前检测 Redis 可达性
- **结果缓存**：模块级 `_REDIS_REACHABLE` 变量，进程生命周期内复用
- **redis-py 8.0 bug**：`socket_connect_timeout` 参数失效 → TCP 探活绕过
- **降级效果**：无 Redis 时启动延迟 < 2ms，每次操作 < 2ms

## 五、开发中遇到的问题与解决

### 1. 服务器启动慢（~15s）
**原因**：sentence-transformers 初始化 BGE 模型时尝试联网检查 HuggingFace 新版本。
**解决**：加载时加 `local_files_only=True`，禁止联网检查。启动时间从 ~15s 降到 ~2s。

### 2. 搜索爬虫 URL 编码报错
**原因**：用户多行输入包含 `\n` 拼接到 URL 不合法。
**解决**：`urllib.parse.quote(query)` 对全部特殊字符做 URL 编码。

### 3. redis-py 8.0 socket_connect_timeout 失效
**原因**：redis-py 8.0 内部实现变更，该参数不再传递给底层 socket。
**解决**：TCP 探活 `asyncio.open_connection(host, port, timeout=2)`，不通直接走内存。

### 4. 向量数据库依赖过重
**解决**：去掉 ChromaDB，改用 BGE 主嵌入 + numpy 矩阵运算 + JSON 索引。N < 1w 条时性能无差异。

### 5. 搜索引擎反爬拦截
**解决**：伪装浏览器 User-Agent + Accept-Language: zh-CN + follow_redirects。

### 6. LangGraph globals() KeyError
**原因**：`build_agent_graph()` 用 `globals()[node]` 动态查找节点函数，Python 3.13 循环导入时部分函数不在 globals 中。
**解决**：改为显式 `graph.add_node("intent", intent_node)` 逐个注册。

## 六、面试 Q&A 速查

**Q: 为什么用 LangGraph 而不是 LangChain LCEL？**
> LangGraph 原生支持有向图 + 条件边 + 循环，适合多 Agent 协作。LCEL 偏向链式调用，无法表达 supervisor 回退。

**Q: 为什么 RAG 不用 ChromaDB？**
> N < 1w 条时内存余弦相似度与专用向量库性能差距可忽略，去掉外部依赖降低部署门槛。

**Q: 流式输出怎么实现？**
> 后端 `ChatOpenAI.astream()` → FastAPI `StreamingResponse(SSE)` → 前端 `fetch ReadableStream` 逐 token 更新。

**Q: ReAct 和 LangGraph 什么关系？**
> ReAct 是决策范式（Thought→Action→Observation），LangGraph 是实现引擎。ReAct 循环逻辑封装在 `_run_react_loop()`，LangGraph 控制节点生命周期和路由。

**Q: Function Calling 怎么实现的？**
> MCP 工具列表自动转换为 OpenAI Function Calling schema（`_build_tools_schema`），传入 `ChatOpenAI(tools=...)`。LLM 返回 SDK 解析的 `tool_calls`，无需正则解析。本地模型不支持时自动降级为文本 ReAct，通过 `Action:` 正则提取工具调用。

**Q: 为什么 RAG 必须走 MCP？**
> 五层架构强制解耦：所有数据源统一注册、统一调用、统一日志。节点只知工具名和参数，不知数据源内部实现。新增数据源只需注册 MCP 工具。

**Q: LoRA 解决什么问题？**
> 让本地小模型（Qwen2.5-3B）在特定领域（世界杯/足球）达到接近大模型的效果。34 条 Q&A 训练 96 秒，永久免费推理，降低 DeepSeek API 调用成本。
